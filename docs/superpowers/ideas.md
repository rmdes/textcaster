## IndieWeb layer — milestone-suite (decomposition, 2026-07-19)

Textcaster today **consumes** microformats (feed discovery ingests h-feed/h-entry/
h-card; reply-context h-cite shipped) but **emits zero mf2** (`grep h-entry|h-feed|
h-card|rel=me web/src` = empty) and speaks none of the IndieWeb protocols. "The
IndieWeb layer" is therefore a **publish/interop gap** and a *suite* of independent
sub-projects — captured here as one milestone theme so the pieces (several already
have their own entries below / in Roadmap stubs) are sequenced, not scattered. Each
gets its own spec → plan → SDD cycle when picked up. **Not started; roadmap only.**

- **A — Emit mf2 ("the page is the feed")** *(foundational)* — mark up our rendered
  HTML: `.timeline`→h-feed, `.post`→h-entry (`e-content`, `u-url`/`u-uid`,
  `dt-published`), byline→nested **h-card** (`p-name`/`u-url`/`u-photo`), replies→
  `u-in-reply-to`; a **representative h-card per user** at `/u/:handle`; optionally an
  instance-level h-card. Detail: [[The page is the feed]] entry below. Self-dogfoods
  (`discoverFeed` could read our own pages) and **unlocks B/C/G**.
- **B — rel=me + h-card verification** — users list "also me" URLs; render `rel="me"`
  on their h-card (from A); Textcaster fetches each and checks the *back-link* →
  "confirmed" badge, decoupled from auth. Detail: [[Confirmed elsewhere]] entry below.
  Extends A's h-cards.
- **C — Webmention (in/out)** — send when a local post replies-to/links an external
  URL advertising a webmention endpoint; receive + thread mentions to local posts.
  Needs A (parseable mf2 sources + discoverable targets). Roadmap stub.
- **D — IndieAuth sign-in** — claim a remote identity by domain ownership, on the
  better-auth foundation. Roadmap stub; conceptually pairs with B's rel=me.
- **E — Micropub (posting-in)** — post to your instance from any Micropub client.
  Roadmap stub. Independent auth/API surface.
- **F — Microsub (reading-out)** — expose the timeline as Microsub channels + JF2 so
  IndieWeb readers (Monocle, Together) read your Textcaster river. Detail:
  [[Microsub read endpoint]] entry below. Independent read-API surface.
- **G — u-syndication links** — "this post also lives at…" (POSSE ledger). Detail:
  [[Syndication links]] entry below. Small add-on to A's h-entry.
- ✅ **Reply-context h-cite (consume)** — SHIPPED 2026-07-19 (below).

**Dependency shape / sequencing:** **A is the foundation** (inverse of what we already
consume; most on-vision; no-JS-native) → **B/C/G build on it** → **D, E, F** are
independent tracks sequenced by appetite. Recommended first when picked up: **A**
(emit mf2 + representative h-cards). Coordinate with the in-flight per-user-feeds
milestone before opening this as a concurrent large thread.

## Live edits over the wire — "posts that stay current"

**Status:** ✅ SHIPPED (2026-07-19, deployed to all 3 instances) — local post
editing (`PATCH /posts/:id`) + ingest edit-detection + `post_revisions` history
(`/post/:id/history`) + `<atom:updated>`/`date_modified` federation + SSE
upsert-by-id with an "edited" marker. Matched the rss.chat feature we lacked:
Dave edits posts *in place* under a stable guid; we used to freeze the body at
first poll. **Consequence for moderation:** the new `post_revisions` table is a
`RESTRICT` FK on `posts` — SP3's hard removal had to clear revisions before
posts (`deletePost` + `deleteUserCascade`), else SQLite refused to delete any
edited post (fixed `8a9f94a`). Original design notes retained below for the record.

**Mechanism.** Ingest currently freezes a remote post's body: a re-polled item
whose `guid` already exists makes `insertPost` return false, and the only update
path (`backfillItemExtras`) `COALESCE`s attribution columns and never touches
`content`/`title`. Change that branch so a same-`guid`, changed-body re-ingest
**updates the stored body, stamps `edited_at`, and emits an `edited` event** on
the existing bus → SSE, so the timeline swaps the text in place (jank-free, like
live prepends) with a quiet "edited" marker. Because remote posts are
pass-through re-emitted, a correction then federates onward A→B→C for free.

**Why novel / why it fits.** Textcasting's *own* contract lists posts as
**editable** (`2026-07-15-textcaster-design.md:53`), yet every editable post is
silently pinned to its first-polled version — including from our headline
interop partner rss.chat, which edits in place. The stable permalink-guid we
already lean on (`feed.ts` `localGuid`, walkable-feeds) is exactly what makes an
edit *detectable*: same id, different body = an edit, not a new item. No spec
covers consume-side edit propagation (threading's "editing" non-goal is about
*threads*; rich-content's is about a *local compose* UI).

**Grounding.** `core/src/domain/ingest.ts:167-171` (the dedup else-branch, only
fires on attribution/url) · `core/src/storage/sqlite.ts:307` `backfillItemExtras`
(COALESCE-only, content/title never written) · the guid-as-identity contract.

**Tradeoff (the real one).** It breaks the timeline's append-only immutability,
which is also a trust boundary: a bad-faith source can bait-and-switch — publish
something benign, let it federate/get shared, then rewrite it under the same
permalink, and every follower's copy silently changes. The honest design choice
is "trust the source's edits + an 'edited' marker" vs. "freeze at first-seen";
edit history/diffs are real added weight. (Note: this is the philosophical
*opposite* of Verified Bylines below — one trusts the source more, the other
less. Decide the trust posture deliberately.)

## Verified bylines — check a relayed post against its origin feed

**Status:** candidate (strong — my top vet pick; a *security* capability).

**Mechanism.** When we consume an aggregate feed (rss.chat firehose, another
instance's `/users/rss.xml`), authorship comes from the RSS `<source url>`
element and is stored as `sourceName`/`sourceFeedUrl` with **no check that the
named feed actually contains the item**. At ingest, an item whose arriving feed
URL differs from its own `sourceFeedUrl` is *relayed*, not first-party →
`origin_verified = null`; a pass on the existing poll cycle fetches that
`sourceFeedUrl` (same SSRF-guarded `fetchFeedBody`), confirms the item's `guid`
is present, and flips the flag. Payoff beyond an SSR "verified" badge:
`adoptOrphans`/`findPostByRef` can **refuse to thread a relayed reply** onto a
local post until its origin vouches — closing a cross-instance byline/reply
spoofing vector.

**Why novel.** The trust layer is empty and *deliberately so*: firehose F-3 pins
attribution to `<source url>` and says "do not add phantom consumption" — it
chose to trust, never verify. Distinct from IndieAuth "claimed remote" (that
verifies a *person controls a domain*; this verifies a *specific post is really
in a specific feed*, per-item, even for unclaimed aggregated authors).

**Grounding.** `ingest.ts:71` `sourceFeedUrl: httpOnly(source?.url)` (wire,
unchecked; tests even feed hostile attribution) · `types.ts:32` already stores
the home feed to fetch · `fetchFeedBody` + `runPollCycle` are the reusable
transport · precedent: one nullable column via `backfillItemExtras`.

**Tradeoff.** One extra fetch per distinct relayed origin (poll-scope + attack
surface). And 50-item feed windows mean an old relayed item may have rolled off
its origin and can never be confirmed — so `null` must mean "unconfirmed,"
never "fake."

## Lantern posts — presence keeps your words lit

**Status:** candidate (poetic; softer — a reading/UX mechanic on existing GC).

**Mechanism.** Guest posts already live exactly as long as the guest keeps
showing up: the idle sweep reclaims anonymous accounts and cascade-deletes their
posts, and "idle" is measured from the latest session `updatedAt`, which every
request refreshes — so *reading* already resets the clock. Make it legible:
expose remaining TTL as `fadesInDays` on `/me` for anon sessions, tell guests
"these words fade in ~N days unless you keep visiting — or register to keep
them," and let a guest card visibly **dim toward its fade date** (server-computed
opacity class, works no-JS). Registering is the permanence upgrade (`onLinkAccount`
re-points the row to a non-anonymous user the sweep then skips forever).

**Why it's uniquely feeds-native.** Because content is render-on-read (no stored
HTML) and every feed is generated per-request, a faded guest evaporates from the
timeline, firehose, and every `source:comments` feed *simultaneously, with zero
purge work* — no tombstone, no cached copy. Permanence is the effort; fading is
the default — the inverse of every mainstream timeline.

**Grounding.** `sweepAnonymousUsers` (`sqlite.ts`) + hourly `sweepLoop`
(`server.ts`) · `auth.ts` `anonTtlDays` / `onLinkAccount` · render-on-read feed
routes.

**Tradeoff.** "Gone from every feed at once" is only true on the origin
instance: a guest post already pushed over WebSub/rssCloud and ingested by a peer
is an ordinary remote row on *their* side, beyond the origin's sweep. The UI must
promise "fades here," not "erased everywhere," or it lies about federation.

## Bring your back-catalog — import a feed as *yourself* (account move between instances)

**Status:** candidate (round 2 — strongest of the round; IndieWeb "own & move
your data").

**Mechanism.** Today every ingested feed becomes a *remote user you follow*. Add
the inverse: on a fresh instance B, a new local account points at instance A's
`/users/<handle>/feed.xml`, and Textcaster ingests those items as **its own
local posts** (`source:'local'`, `authorId = me`). Because a local feed already
emits `source:markdown` beside the rendered HTML, the importer reads the raw
markdown back into `content` (lossless — no HTML round-trip) and **preserves each
post's permalink `guid`** (the walkable-feeds thread key), so every reply anyone
made still threads onto the imported copy. Follow graph comes over via the
existing OPML import. The feed *is* the export format — no proprietary blob.

**Why novel.** Not in any spec: the following spec's OPML import makes *remote*
users and its "case 2" (`localHandleForUrl`) exists precisely to *avoid*
self-shadowing — claiming a feed as *your own posts* has no home. Distinct from
IndieAuth "claimed remote" (domain ownership) and from better-auth's deferred
"merge UI" (`2026-07-17-better-auth-design.md:143`, account *merge*, not
cross-instance *move*).

**Grounding.** `feed.ts:68` `itemContentFields` emits `source:markdown` →
lossless re-import · `ingest.ts` parses `sourceNs.markdown`, `insertPost`
idempotent by guid · `feed.ts` `localGuid` = permalink-as-identity · `opml.ts:69`
`localHandleForUrl` already recognizes our own minted feed URLs.

**Tradeoff (a genuine either/or).** A post's identity and its live URL are the
same string. Keep the old permalink guid → history stays threaded but old links
point at instance A and may rot; mint a fresh guid at B → links live but every
existing reply orphans. Can't have both; pick history-threaded vs links-alive
(optionally advertise the new home as a `source:` alternate).

## River feeds — turn the "followed" lens into a subscribable feed URL

**Status:** candidate (round 2 — novel, but crosses the no-republishing line;
scope carefully).

**Mechanism.** The *followed* lens (`/timeline?followed_by=<handle>`) is
currently only a web view + client-side SSE filter. Promote it to a real feed:
`GET /users/:handle/timeline.xml`, built by handing
`getTimeline(FEED_LIMIT, undefined, { followedBy: user.id })` to the existing
generic `renderFirehoseRss()` — the firehose is this route with the filter
removed, and it already stamps per-item `<source url>`. Unlock:
**follow-the-follower** — instance B subscribes to Alice's one river and receives
everyone Alice reads, per-item-attributed, without subscribing to each source
(the live-stream equivalent of importing Alice's OPML, but as one feed that keeps
updating). Following a person's *taste*, not a list of URLs.

**Why novel.** The following spec defines lenses as web views + SSE filters + a
static OPML export — never a merged live river feed. Inverse of "Follow the
source" (that de-aggregates one author *out*; this re-aggregates a reading list
*in*).

**Grounding.** `web/src/lib/lens.ts` (browser-only filter) · `app.ts:335` the
`followed_by=` JSON filter · `app.ts:237` firehose = unfiltered
`getRecentLocalPosts` → `renderFirehoseRss` · `feed.ts:122` `renderFirehoseRss`
is generic over any `TimelineEntry[]` and emits per-item `<source>` — near-zero
new render code.

**Tradeoff.** A river **republishes remote items under this instance's URL** —
crossing the line `resolveFeedUser` deliberately draws (`app.ts:239`
"Pass-through, not republishing"). It makes the instance a relay, inheriting the
stale-copy/byline-trust problems Live-edits and Verified-bylines circle. Safe
scoping: river-over-*local*-posts only (like the firehose); the powerful
cross-instance river means owning the relay's provenance duty.

## Archive feeds — give the window a past (RFC 5005 paged feeds)

**Status:** candidate (round 2 — reopens a *deliberately* deferred call; clever
federation payoff).

**Mechanism.** Feeds emit only the newest 50 items then stop, though the DB keeps
everything. Add an optional `before` cursor to the feed routes so
`GET /users/alice/feed.xml?before=<cursor>` returns the next 50 older posts, and
emit an RFC 5005 `<atom:link rel="prev">` per page (+ `rel="current"` on the
head). The back-catalog becomes a chain of ordinary feed documents any reader —
or threadwalker — can crawl backward, zero new storage, no JS.

**Why novel / caveat.** The feed-output spec-review explicitly decided *"feeds
are windows, not archives"* (`2026-07-15-feed-output-spec-review.md:157`) — so
this **reopens a conscious deferral** rather than finding an unseen gap.
Orthogonal to walkable-feeds (that walks a conversation tree sideways by guid;
this walks a feed's history backward by time). Federation payoff: if an origin
pages its history, a consumer can walk `rel="prev"` back and **re-fetch a
rolled-off item** the origin's head window forgot — partly dissolving the
50-item-window gap Verified-bylines names as permanently `null`.

**Grounding.** `app.ts:221` `FEED_LIMIT = 50`, feed routes pass only a limit
(`:265,281`) while `/timeline` already keyset-paginates with `parseCursor`
(`:335`) · `sqlite.ts:233` `getPostsByAuthor` is limit-only but rides
`posts_author_pub_idx (author_id, published_at, id)` (`:512`) — the index makes
paging nearly free · remote users stay 302 pass-through, so no republishing.

**Tradeoff.** RFC 5005's prize is cacheability (fetch an archive page once,
never again) — but that needs *immutable page boundaries*, and pages are ordered
by mutable `published_at` (editable/backdated posts shift boundaries). Pick:
`rel="prev"` **paged** (honest, no cache promise, readers re-crawl) vs `fh:archive`
**archived** (cache-forever) re-keyed on immutable `created_at` (but display order
can drift from the `published_at` head). Cacheable *or* published-order-faithful,
not both.

## Conversations as outlines — `GET /post/:id/thread.opml`  *(Dave-lens round)*

**Status:** candidate (Dave-lens — strongest of the round; the most
quintessentially Winer idea: outliners + OPML).

**Mechanism.** A Textcaster thread is already a tree — `getThread` resolves the
flat replies into one (keyed on `inReplyToPostId`, orphans hung at root), and the
reply "wedge" redraws it in the browser every page load. But the only wire export
(`renderCommentsFeed`) is a *flat* RSS list that throws the nesting away. Add a
route that emits the thread as a **nested OPML outline**: root post = top
`<outline>`, each reply nested under the one it answers, each node carrying
`text`="author — first line", `type="link" url=<permalink>`, `created=<pubDate>`,
and the `source:markdown` as a `_note`; `<head><expansionState>` mirrors the
wedge's open branches. The OPML *is* the outline you read — openable in Fargo /
Drummer / Little Outliner, collapsible, republishable.

**Why novel.** OPML-as-blogroll is built (following spec, flat follow-list — its
import even *discards* folder structure); OPML-as-source-filter is roadmapped
(README); OPML-as-*conversation-document* has no home anywhere. Threading/
walkable-feeds emit conversations only as flat `source:comments` RSS. The gap:
Textcaster rebuilds the outline on every page load and never lets you take it.

**Grounding.** `sqlite.ts` `getThread` → `orderThread` (resolved tree) ·
`web/src/lib/wedge.ts` `childrenOf`/`subtreeIds` + `ReplyTree.svelte` (same tree,
drawn) · `feed.ts:192` `renderCommentsFeed` (today's flat export) · `opml.ts`
already uses feedsmith `generateOpml` (nested `outlines` + `_note` + head
supported).

**Tradeoff.** OPML is a strict tree, but Textcaster *honestly orphans* replies
whose parent never resolved (hung at root). The export must drop or "dangling"-
mark those — it's faithful to what the instance *resolved*, not to the
cross-instance conversation.

## Linkblog posts — the headline points out, the permalink stays home  *(Dave-lens round)*

**Status:** candidate (Dave-lens — very Winer; linkblogging *is* Scripting News).

**Mechanism.** Add an optional outbound target to a local post (nullable
`link_url` = the thing you're pointing at). In the feed, emit `<link>` = that
external URL while `<guid>` stays your permalink (still the thread key), with your
commentary in `<description>`/`source:markdown`. Mark it with a Textcaster-
proposed `source:linkblog` element so a peer renders it as a *link post*, not a
post whose permalink merely happens to live offsite. No-JS: the server-rendered
headline is just an `<a href>` pointing out.

**Why novel.** Walkable-feeds *deliberately* makes `<guid>` == `<link>` ==
reply-ref share one source (`p.url`) so they "can never drift" (spec:51).
Linkblogging is precisely the intended *exception* — the one case `<link>` should
diverge from the permalink — and no spec contemplates it. `grep` confirms no
`link_url`/`linkblog` concept exists anywhere; `Post` only knows `url` = permalink.

**Grounding.** `feed.ts:20` `localGuid` + emission — `<link>` and `<guid>` both
come from `p.url`; that conflation is what to split · `types.ts` `Post` (one new
nullable field) · `ingest.ts` derives an item's `url` from `it.link ?? guid`, so a
divergent `<link>` flows in · walkable-feeds spec:51 (the "never drift" pin this
excepts). Protocol: RSS 2.0 `<link>` is canonically "the item's URL" — for a link
post the item *is* the link.

**Tradeoff.** Once `<link>` ≠ permalink, a plain reader's "open" leaves your site
(the point of linkblogging — and what walkable-feeds designed against). On
ingest, a linkblog `<link>` becomes the consumer's stored `url`, and
`findPostByRef`'s url-arm needs global uniqueness — so two people linkblogging the
same article collide on that URL and any *url-keyed* reply orphans. Threading
survives because it keys on the permalink **guid**, which makes the
`source:linkblog` marker load-bearing (else a link post is indistinguishable from
an offsite-permalink post).

## Tail the river — a public SSE firehose you `curl`, no callback required  *(Dave-lens round)*

**Status:** candidate (Dave-lens — cheap + clean, but the most incremental:
public-exposes the existing SSE stream).

**Mechanism.** `GET /users/river` — a public, no-auth `text/event-stream` that
pushes every new *local* post the instant it lands: the live wire the web UI
already rides, promoted to a first-class open endpoint. Reuses `streamSSE` +
`bus.onNewPost` + Last-Event-ID replay verbatim, adding one filter
(`entry.source === 'local'`) and the same `<source url>` attribution the RSS
firehose carries, one JSON object per frame. Pitch: `/users/rss.xml` is the
firehose you *poll*; `/users/river` is the same firehose you *tail* — no WebSub
hub, no rssCloud callback server. `curl -N https://instance/users/river` is a
complete realtime reader.

**Why novel.** The firehose spec specs it only as RSS 2.0 + WebSub/rssCloud —
never an SSE surface. Distinct from `ideas.md` "River feeds" (that promotes the
*followed lens* to a *pull* RSS feed; this is the unfiltered *public push* stream)
and from the existing `/timeline/stream` (private, web-proxy-only, full
local+remote timeline). Dave built rssCloud to kill polling, but rssCloud/WebSub
still make *you* run a callback server — SSE is that instinct minus the callback.

**Grounding.** `app.ts:371` `/timeline/stream` (`streamSSE`, `bus.onNewPost`,
replay via `getTimelineAfter`/`REPLAY_CAP`) — the machinery to reuse ·
`ingest.ts:166` emits *remote* posts onto the same bus as `service.ts` emits
local, so the endpoint must filter to `source==='local'` (same boundary
`push.ts:200` `onLocalPost` enforces).

**Tradeoff.** SSE is connection-bound and push-only — no *durable* subscription
that survives the client being offline. Miss the window (beyond the bounded
replay) and posts are gone from the stream; you fall back to the RSS firehose to
catch up. It's for watching live, complementing (not replacing) the pull/push
feeds.

## Reply-context that rides along — consume the embedded `h-cite` in `in-reply-to`  *(Aaron-lens round)*

**Status:** ✅ SHIPPED (2026-07-19, on `main`, not yet deployed) — parse h-cite →
thread ref (fixed the orphan bug) + author-gated author/snippet context, trust-gated
(nulled once resolved) + plain-text render on 4 surfaces. Part of the **IndieWeb
layer** milestone-suite (consume side of A). Original writeup retained below.

**Mechanism.** When Textcaster ingests an h-feed, `discovery.ts` reads each
entry's `in-reply-to` **only if it's a bare URL string**; if it's an *object* (an
embedded `h-cite` — the IndieWeb reply-context carrying the parent's author,
name, and a content snippet, which Aaron publishes on every reply), it's thrown
away → `null`. Instead: (a) pull the h-cite's `.url` as the thread ref so the
reply *threads* instead of orphaning, and (b) keep its `author`/`name`/`content`
as stored reply-context so an unresolved reply renders "In reply to *@aaronpk* —
'…the line…'" instead of floating mutely at the root. XRay applied at ingest —
the reply describes its own parent, no fetch.

**Why novel.** Two-for-one: fixes a **latent threading bug** (object-form
replies orphan even though their URL is right there) and adds reply-context. It
threads the threading spec's exact needle — its non-goal is *"fetching reply
context for unresolved refs"* (`2026-07-16-threading-design.md:233`); this
context arrives **inline in the reply's own mf2**, zero fetch, so it delivers
what the spec refused (context) without the thing it refused (fetching). `grep
repost|rsvp|reply-context|h-cite` = zero.

**Grounding.** `discovery.ts:55` the literal discard
(`typeof irt[0] === 'string' ? irt[0] : null`) · `@paulrobertlloyd/mf2tojf2`
(already a dep) emits `u-in-reply-to h-cite` as `{ url, name, author }` · the
`.url` flows straight into existing `findPostByRef`/`adoptOrphans` resolve-once
(`sqlite.ts:252,275`) — no new threading logic · one nullable column for the
context, like `sourceName`/`contentMarkdown` · honest-orphaning is already the
house style → a legible orphan instead of a mute one.

**Tradeoff.** Reply-context is the *replier's own unverified claim* about the
parent (the Verified-bylines trust seam) — a bad-faith reply can embed a
fabricated snippet, and when `adoptOrphans` later resolves the real parent it may
contradict. Honest posture: render embedded context only while unresolved, label
it "as quoted by …", and let the real post win the instant it lands.

**Status:** candidate (Aaron-lens — most conceptually striking; his signature
protocol, and Textcaster already *is* the server).

**Mechanism.** Add `GET /microsub` speaking the Microsub *read* surface:
`?action=channels` returns the lenses as channels (`global`→firehose,

the roadmap (which lists IndieAuth/Micropub/Webmention) — so it dodges the
roadmapped-three exclusion while being Aaron's signature protocol. Distinct from
"River feeds" (RSS-over-a-lens) and "Tail the river" (public SSE) — this is
channels + JF2 + paging, a *read protocol* existing reader clients speak.

**Grounding.** `service.ts:59` `getTimeline(limit, before?, filter?)` + `:103`
`getRecentLocalPosts` — cursor-paginated channel backends already exist ·
`app.ts:359` `/timeline` already resolves `followed_by`/`author` + emits keyset
`nextCursor` (= Microsub `paging.before`) · dual html+markdown → JF2
`content:{ html, text }`; `inReplyTo`→`in-reply-to`; `author`→JF2 card ·
`@paulrobertlloyd/mf2tojf2` already a dep (no new dep). Protocol: Microsub
`channels`/`timeline` + JF2 (Aaron Parecki / W3C Microsub CG).

**Tradeoff.** Microsub channels are conventionally *private, per-user, mutable
inboxes* (clients expect `mark-read`, unread counts, add/remove feeds through the

inboxes"). Ship the honest read-only subset now (query actions over public
lenses) — a client sending `method=mark_read` gets unsupported-action — vs. a
per-user channel-state table (new storage, real weight) for a full inbox server.

**Status:** candidate (Aaron-lens — the most iconic Aaron primitive, but thinner
standalone).

**Mechanism.** Add nullable `syndicationUrls: string[]` to `Post` (the external
URLs where a local post was *also* published — a Mastodon toot, a blog permalink).

**Why novel.** `grep u-syndication|rel=syndication|source:syndication` = zero —
feeds carry `self`/`hub`/`source` links only. The single most iconic Aaron
primitive (he coined `u-syndication`), simply absent. Distinct from "Linkblog"
(`<link>` points *out* to a thing you discuss) — this points *sideways* to copies

**Grounding.** `types.ts:18` `Post` (one nullable field, like `sourceFeedUrl`) ·

analog) · `feed.ts:80-116` `renderRssFeed` per-item `atomLinks` = emission site ·

## The page is the feed — emit the microformats2 you already parse  *(Tantek-lens round)*

**Status:** candidate (Tantek-lens — most Tantek-authentic; mf2 is his invention,
"the page is the data" is his philosophy verbatim).

**Mechanism.** Textcaster *consumes* mf2 (feed-discovery ingests h-feed pages)
but its own rendered HTML carries **zero** microformats. Mark up the three
server-rendered templates with the same vocabulary `discovery.ts` reads: the
`.timeline` `<ul>` → `h-feed`, each `.post` → `h-entry`, the byline author → a
nested `h-card` (`p-name` + `u-url` + avatar `u-photo`), `.body` → `e-content`,
the permalink → `u-url u-uid`, `<time datetime>` → +`dt-published`. On the
conversation page, each reply `h-entry` gains `u-in-reply-to` (parent permalink),
so `/post/:id` becomes a self-describing mf2 thread any parser (XRay, Parse-This,
another Textcaster) walks with **no feed, no threadwalker**. The loop closes on
itself: `discoverFeed()` — which ingests everyone else's h-feeds — now reads a
Textcaster page right back (selfdogfood as a literal round-trip).

**Why novel.** `grep h-entry|h-feed|h-card|rel=me web/src/` = empty; mf2 is
consume-only (README: "ingests h-feed / microformats2 pages"). Distinct from the
Aaron round: "Reply-context" *consumes* embedded h-cite (this *emits*); "Microsub"
is a JSON/JF2 read *API* (this is HTML, no new route); "Syndication links" points
sideways. No spec/idea contemplates Textcaster's own HTML carrying mf2.

**Grounding.** absence: `grep …web/src` empty · consume mirror:
`discovery.ts:22-59` (h-feed→h-entry, `e.url`, `e.published`, `in-reply-to`) +
`ingest.ts:256` ("the page is the feed") · emit sites: home `+page.svelte:90`,
`u/[handle]/+page.svelte:79`, `post/[id]/+page.svelte:82`, `ReplyTree.svelte`
(already-nested replies = the `u-in-reply-to` site), `PostBody.svelte` (the
sanitized `{@html}` chokepoint = a clean `e-content`).

**Tradeoff.** Canonical-URL duplication: the same post becomes an `h-entry` on up
to four pages (home, author page, conversation root, nested-in-wedge), all sharing
one `u-url` — a naive crawler sees it repeated and must dedupe by `u-url` (well-
behaved parsers do). Emit `u-in-reply-to` from *every* h-entry with an
`inReplyTo`, on every page, so the relationship survives whichever page a parser
lands on — at the cost of markup redundant with the DOM nesting.

## Handle history — your old address keeps working  *(Tantek-lens round)*

**Status:** candidate (Tantek-lens — durability; *also fixes a real defect*:
rename breaks every link + opens impersonation).

**Mechanism.** *Cool handles don't 404.* When a local user renames (`PATCH /me`,
`alice`→`alice-smith`), the old handle silently dies today: `/u/alice`,
`/users/alice/feed.xml`, and every already-federated `<source url=…/users/alice/…>`
attribution all 404, and the freed handle is instantly claimable by an
impersonator. Fix: on rename, record the old handle in a `handle_aliases(handle
PK, user_id)` table; `getUserByHandle` falls back to it; when a handle resolves
*via an alias*, respond **301 Moved Permanently** to the same path under the
current handle (profile + feeds), plus `rel="canonical"`/`atom:link rel="self"`
on the live feed. Old handle stays reserved forever (anti-impersonation); links,
subscriptions, and federated attributions survive; RSS readers auto-repoint on
the 301.

**Why novel.** `grep redirect|canonical|alias|301|rename` across all specs:
`better-auth-design.md:145` specs the rename mechanism but says nothing about the
old handle surviving; `feed-output` covers only the *remote* 302 pass-through.
Distinct from "Bring your back-catalog" (cross-instance *move*) and "Archive
feeds" (time-paging). No entry touches rename/redirect/canonical/alias.

**Grounding.** `sqlite.ts:122` `updateUserProfile` overwrites `users.handle` in
place; `handle text NOT NULL UNIQUE` (`:454`) → old name immediately re-claimable ·
`app.ts:111` `resolveUser` / `:234` `resolveFeedUser` resolve *only* via
`getUserByHandle` → old handle → 404 · contrast: `app.ts:239` remote pass-through
is **302 because feedUrl is mutable**; a local rename alias points at the
immutable `users.id` UUID, so **301 is the honest choice**. Post permalinks
(`localGuid` = `/post/:id`, UUID) are already rename-proof — the identity URL is
the one gap.

**Tradeoff.** Reserving every retired handle forever means the namespace only
shrinks — a handle-cycler squats each old name permanently, and an
abandoned-then-renamed handle can never be reissued. Alternative: expire aliases
after N days (trades link durability for namespace reuse). Can't have both
permanent links and a recyclable namespace; Tantek's answer is keep the URL alive.

## Confirmed elsewhere — self-verifying representative h-card (bidirectional `rel="me"`, zero login)  *(Tantek-lens round)*

**Status:** candidate (Tantek-lens — rel=me/h-card/XFN are his; clever, but
nearest the roadmapped IndieAuth's shadow).

**Mechanism.** Give each local user a small "this is also me" URL list (Mastodon,
personal site, GitHub). Turn `/u/:handle` into a real **representative h-card**
(`<h1>` → `h-card` with `p-name`/`u-url`; each identity URL as
`<a rel="me" class="u-url">`). The surprise: Textcaster **checks the back-link** —
reusing `discovery.ts`'s rel-urls machinery over the SSRF-guarded `fetchFeedBody`,
it fetches each claimed URL and asks "does this page link `rel="me"` back to my
profile?" Confirmed if yes, unverified claim if not. Tantek's two-way `rel="me"`
consolidation, used purely for *display*, decoupled from any auth login.

**Why novel.** The founding design mentions `rel="me"` + h-card only *inside* the
roadmapped IndieAuth **auth** flow (`2026-07-15-design.md:31-32,48`). The novel
move is separating rel=me *verification* from *auth* — a pure identity-display
verifier, shippable before IndieAuth and the substrate IndieAuth later reuses.
Distinct from "Verified bylines" (that verifies a *post* is in a *feed*; this
verifies a *person's identity links agree*). No spec/idea has it.

**Grounding.** `discovery.ts:33-34` already iterates `rel-urls` with
`rels.includes('alternate')` — the back-link check is the same loop with
`'me'` · `ingest.ts:179` `fetchFeedBody`/`checkCallbackUrl` (SSRF-guarded, each
redirect hop re-validated) reused verbatim · `server.ts:65` sweep loop = the
re-verification cadence · `types.ts:8` `User` has no links field → one nullable
`links` JSON column (`[{url, verified}]`) · `u/[handle]/+page.svelte:69` the `<h1>`
= h-card root · `settings` + `PATCH /me` = the edit surface.

**Tradeoff.** A "confirmed" badge is a snapshot of a *symmetric link*, not an
identity proof — bidirectional `rel="me"` proves only that one entity controls
both endpoints, never *who* they are (no authority — the point). The back-link
can be added to pass, then removed; re-verification lags by one poll interval, so
"confirmed" can be briefly stale. Label it "these links point at each other," not
"verified human," or it overpromises; each link costs a periodic outbound fetch.

## Stream guardrails — cap concurrent SSE per IP + a max connection lifetime  *(hardening)*

**Status:** candidate (hardening, low urgency — measured *zero* accumulation in
prod today; record so it isn't rediscovered under load).

**Mechanism.** `/timeline/stream` is a public, unauthenticated `text/event-stream`
with **no per-IP concurrency cap and no max lifetime** — a connection lives until
the client drops. Cleanup is already correct (`stream.onAbort(off)` removes the
bus listener; the 15s ping loop exits on `stream.aborted`), so this is purely
*bounding*, not fixing a leak. Add either/both: (a) a per-IP concurrent-connection
counter that rejects/queues past N; (b) a max SSE lifetime that closes the stream
after M minutes — `EventSource` auto-reconnects (replaying via `Last-Event-ID`),
which also naturally sheds zombie/half-open holders. Reuses the existing
`streamSSE`/`bus`/replay machinery; no new transport.

**Why it fits.** `/stream` is public by design (the live public timeline — bots
hit it like any public URL). Today they hit-and-leave: a live check showed **1**
held connection into core, core RSS flat at **~150 MB**, no accumulation. But
nothing bounds a *deliberate* connection-exhaustion actor opening thousands of
concurrent streams to tie up sockets + listeners. The `event: ping` heartbeat a
viewer sees on the raw URL is the intended keepalive, not the concern.

**Grounding.** `core/src/api/app.ts:371` `/timeline/stream` (`streamSSE`,
`bus.onNewPost`, `onAbort(off)` cleanup at `:377`, 15s ping at `:398`) — no
rate-limit or cap on this route · `cloudron/nginx.conf` routes `= /stream` → web
→ core, publicly exposed · replay contract (`Last-Event-ID` → `getTimelineAfter`
/ `REPLAY_CAP`) makes a forced reconnect cheap and lossless.

**Tradeoff.** A hard per-IP cap false-positives on NAT'd / shared-IP viewers
(many legit readers behind one address); a max-lifetime forces periodic reconnect
churn (a brief gap + the replay cost each cycle). Tune conservatively — only worth
building if bot/abuse volume actually climbs; premature caps degrade honest
viewers for a threat not yet observed.

## better-auth plugin & adapter surface (audit 2026-07-19)

Audit of the better-auth building block against its plugin catalogue. **In use
today** (`core/src/auth.ts`): `emailAndPassword` (hard verify), `magicLink`,
`anonymous`, plus rate-limiting and x-forwarded-for IP handling. The items
below are *available* but not adopted — each is a real feature (schema and/or
UI and/or the load-bearing web proxy), so each goes through brainstorm→spec, not
a drop-in `plugins: [...]` add. Consult the `better-auth` MCP for current API.

- **passkey (WebAuthn)** *(⭐ promoted 2026-07-19 — brainstorm queued)* — passwordless
  login via platform/roaming authenticators. **Grounding:** better-auth ships
  `passkey()`; we already have the emailed-link proxy that would relay its
  ceremonies. **Why it fits:** phishing-resistant second/primary factor on top
  of hard email verify. **Tradeoff:** adds `passkey`/`credential` tables
  (schema migration), needs client-side `startRegistration`/`startAuthentication`
  wiring in `web/` (JS-required flow — must degrade, per no-JS-first invariant),
  and the proxy must pass the WebAuthn JSON through. Not a config toggle.

- **username** *(candidate)* — login by username instead of/alongside email.
  **Tradeoff:** Textcaster already has a `handle` on the core user row; a
  better-auth `username` field would be a *second* name to keep in sync with
  `handle`. Likely reject unless we want handle==auth-username unified — spec it
  before touching, to avoid two sources of truth.

- **multi-session** *(⭐ promoted 2026-07-19 — brainstorm queued)* — hold several signed-in accounts in one
  browser, switch without re-login. **Tradeoff:** interacts with the anonymous
  guest carry-over (`onLinkAccount`) and the cookie relay in
  `web/src/lib/server/session.ts` — both assume one active session. Non-trivial;
  needs its own brainstorm on how guest-upgrade behaves with N sessions.

- **open-api** *(⭐ promoted 2026-07-19 — brainstorm first)* — serves an OpenAPI schema + a
  reference UI for the better-auth routes. **Why it fits:** core is
  internal/headless; a generated auth-route reference aids the web-proxy work.
  **Tradeoff:** exposes an auth-surface doc endpoint — must stay behind the
  internal boundary (never through Caddy's public allowlist). Lowest-risk of the
  four; still verify it doesn't widen the public surface.

- **`@better-auth/mongo-adapter`** *(deferred — YAGNI today)* — swap the
  better-sqlite3 store for MongoDB. **Grounding:** the "future Cloudron→Mongo"
  idea. **Reality:** Cloudron currently ships **SQLite/localstorage, not Mongo**
  ([[cloudron]] memory); the shared `sqlite` handle from `repo.raw` is
  load-bearing across auth + the domain DB. Switching auth to Mongo would split
  the datastore (auth in Mongo, feeds/posts in SQLite) unless the *whole* repo
  migrates. Not worth adding the dep until a concrete Cloudron-Mongo requirement
  exists; revisit only then.

- **SvelteKit official integration** *(likely reject)* — better-auth's stock
  SvelteKit client/handler. **Reality:** we deliberately hand-roll the
  `/api/auth/[...path]` proxy because emailed GET links carry no `Origin` and
  better-auth 403s them (CLAUDE.md invariant + `web/.../[...path]/+server.ts`).
  The stock integration doesn't inject `Origin` on native link-clicks, so
  adopting it would regress verified/magic-link flows. Keep the proxy; only
  adopt stock pieces that don't touch the emailed-GET path.
